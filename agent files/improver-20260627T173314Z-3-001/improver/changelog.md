## 2026-06-25 — Diagnosed stock `opencode` Bun segfault on `--continue` (no fix applied; user to choose workaround)
- **Symptom (user-reported)**: `opencode` (standalone Bun exe at `C:\Users\narea\.bun\bin\opencode.exe`, v `0.0.0-dev-202606250443`) sometimes panics with `panic(main thread): Segmentation fault at address 0x225BAC36AA2` ("oh no: Bun has crashed. - **Investigation**:
  - `~/.local/share/opencode/opencode.db` = **409 MB** (429,035,520 bytes) + **4.2 MB uncommitted WAL** (`opencode.db-wal`, 4,165,352 bytes), last written 2026-06-22. WAL has not been checkpointed in 3 days.

  - Bun features at crash: `workers_spawned(2)`, `process_dlopen(3)`, `tsconfig_paths`, `standalone_executable`, `transpiler_cache(5)`.
  - Crash address `0x225BAC36AA2` is in the TB+ range — memory corruption / use-after-free signature, not a clean NULL deref.
- **Strong correlate (high confidence)**: the 409 MB DB + 4 MB uncommitted WAL. Combo of SQLite WAL-recovery + 409 MB scan + worker spawning + 3 native FFI modules (opentui-c5en9p2g.dll, sqlite-better, etc.) is exactly the surface where intermittent FFI/binding segfaults show up.
- **Workarounds offered to user (none applied — user choice)**: (1) skip `--continue`; (2) try `OPENCODE_DISABLE_WORKERS=1 opencode`; (3) prune/vacuum the bloated DB; (4) file a Bun bug at `https://bun.report/1.3.14/...`.
